## Overview

Production-grade Meta Pixel + Conversions API (CAPI) tracking plugin for OctoberCMS 4.x + Lovata Shopaholic ecosystem. Server-deduplicated event_id pipeline (browser `fbq()` and server `POST /events` share same UUIDv4 + event_time). Closed `_fbp`/`_fbc` empty-cookie bug + attribution gap on bank-transfer and admin-marked-paid orders.

Shopaholic-coupled implementation. Tight binding to `Lovata\OrdersShopaholic\Models\Order`. Multi-site via Order.site_id (Lovata-authoritative column). Plugin owns its idempotency table; never mutates upstream schema.

**Pivot trigger (2026-05-15):** decision to pursue marketplace-grade plugin sellable to any OctoberCMS operator regardless of cart-plugin. v1.x architecture too tightly coupled to Shopaholic to extend cleanly — restart in v2.0 milestone with adapter pattern (`EventSubjectAdapter` + `ValueResolver` interfaces; per-cart adapters: Shopaholic, MelonCart, Mall, ThemeAction). See `project-metapixel-v2-reset` memory.

**Key Decisions (locked, carry forward to v2.0):**

- **event_id direction = server → frontend** only. Meta dedupes on event_id match within ±10s window. Never reverse.
- **CR-02 host-spoofing fix** via TrustedHosts allowlist pattern. Naive `count(explode('.', host)) - 1` is exploitable + wrong for multi-TLD. v1.x hardcoded HOST_INDEX_MAP; v2.0 will operator-supply via Settings + php-domain-parser for index derivation.
- **CR-03 fbclid validation** = `[A-Za-z0-9_-]` charset, ≤255 chars, invalid → skip `_fbc` (fail-safe).
- **content_ids format** = `SKU-{product_id}[-{offer_id}]` to match Facebook Catalog feed exporter. No alternative format.
- **Paid-status trigger** = `new-payment-received` (Status ID=5, custom). Configurable dropdown.
- **Idempotency** = plugin-owned `logingrupa_metapixel_event_log` UNIQUE race-fence on `(subject_type, subject_id, event_name, channel, site_id)`. Plugin never mutates upstream schema. v2.0 carries this pattern forward verbatim.
- **Multi-site site_id source** = Subject model attribute (Order.site_id for Phase 3.1; v2.0 generalizes to `EventSubjectAdapter::getSiteId()`).
- **Boot-time missing pixel_id** = `Log::warning` + plugin-disabled flag (NOT throw). Cascading boot failure would nuke entire site.
- **No `assert()`** anywhere — prod `zend.assertions=0` silently no-ops. Enforced by `spaze/phpstan-disallowed-calls`.
- **Settings extends `Lovata\Toolbox\Models\CommonSettings`** — ecosystem norm, auto-caches, Multisite-trait-compatible.
- **EventLogWriter::record fail-safe** = false on UNIQUE collision OR DB failure. Caller does NOT proceed to HTTP POST / fbq fire on false.

**Architecture pivot rationale (2026-05-15):**

User goal evolved from "make plugin resellable to other shops on nailscosmetics ecosystem" to "marketplace plugin sellable to any OctoberCMS operator regardless of cart-plugin choice." v1.x is too Shopaholic-coupled to extend cleanly:
- `OrderStatusWatcher` subscribes Shopaholic-specific event
- `PayloadBuilder` derives content_ids from Shopaholic Offer/Product
- `UserDataHasher` reads Shopaholic Order → User relation
- `SiteResolver::forOrder(Order)` type-hints Lovata Order
- `PurchasePixel` routes via Order.secret_key
- `MetapixelTestCase::bootOrdersTable` boots Shopaholic Orders table
- `composer.json` requires `lovata/shopaholic-plugin` + `lovata/ordersshopaholic-plugin`
- Plugin name itself: `Logingrupa\Metapixelshopaholic`

v2.0 introduces:
- Namespace rename: `Logingrupa\Metapixel` (drop "shopaholic")
- Directory rename: `plugins/logingrupa/metapixel/`
- Adapter pattern: `EventSubjectAdapter` + `ValueResolver` interfaces
- Per-cart adapters: Shopaholic (port v1.x logic), MelonCart, Mall, ThemeAction (generic theme events)
- `composer.json`: `lovata/shopaholic-plugin` becomes `suggest:` not `require:`. Auto-detect at boot.
- PHP support broadens: `"php": "^8.3 || ^8.4"` (v1.x was 8.4-only)
