<objective>
Ship `PluginGuard`: the single source of truth for whether the plugin is operationally disabled (missing `pixel_id`). It is a `October\Rain\Support\Traits\Singleton`-backed helper that reads Settings once on `init()`, logs a warning when `pixel_id` is empty, sets an internal `$bIsDisabled = true`, and exposes the result both as a static API (`PluginGuard::instance()->isDisabled()`) and as a container singleton (`App::make('metapixel.disabled')`) for short-circuit use by Phase 3+ event handlers.

Purpose: The "boot-time missing pixel_id = log + disabled flag, NOT throw" decision (PROJECT.md Key Decisions row 5; SKEL-05) protects upstream plugins (Campaigns, PromoMechanism, Order) from cascade failure. PluginGuard centralises that policy so no handler reinvents the check. The container-singleton bridge gives consumers a one-liner short-circuit (`if (App::make('metapixel.disabled')) { return; }`) — documented in the helper's PHPDoc as the contract for Phase 3+.

Output: `classes/helper/PluginGuard.php`, modified `Plugin::boot()` (single prime call), modified `MetapixelTestCase::tearDown()` (PluginGuard flush hook), and `tests/Feature/BootsWithoutPixelIdTest.php` covering the three SKEL-05 invariants. Plan 02-03 adds middleware on top; this plan is purely the boot-safety helper.

Directory casing note: this plan creates `classes/helper/PluginGuard.php` (all-lowercase directory segments). Sibling-plugin precedent: `plugins/logingrupa/goodsreceivedshopaholic/classes/support/`, `plugins/lovata/toolbox/classes/helper/UserHelper.php`. PSR-4 namespace `Logingrupa\Metapixelshopaholic\Classes\Helper\PluginGuard` resolves via Composer's case-insensitive PSR-4 autoloader in this environment per the sibling-plugin precedent — verified at autoload time by Task 1's `class_exists` check.
