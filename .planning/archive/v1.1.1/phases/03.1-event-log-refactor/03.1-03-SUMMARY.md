## v1.1.0 SCE-03 BREAKING CHANGE ‚Äî Side-by-Side Constructor Signatures

```php
// Phase 3 (Plan 03-05, SCE-03 locked):
public function __construct(
    public readonly string $sEventName,
    public readonly array $arPayload,
) {}

// Phase 3.1 (Plan 03.1-03, REFAC-06 ‚Äî v1.1.0 minor bump):
public function __construct(
    public readonly string $sEventName,
    public readonly array $arPayload,
    public readonly Order $obSubject,    // NEW ‚Äî required for EventLogWriter::record subject param
) {}
```

The break is documented via three artifacts:
1. **`updates/version.yaml`** entry for v1.1.0 (Wave-1 deliverable) names the phase + the schema migration; the constructor break is implicit in the plugin's "no backward compatibility" Phase 3.1 charter (BRIEF.md line 77).
2. **Threat model T-3.1-12** ("Tampering, SendCapiEvent constructor break") is `accept`-disposition with rationale: "No code outside this plugin instantiates SendCapiEvent."
3. **`test_constructor_requires_order_subject_parameter`** asserts `ArgumentCountError` / `TypeError` on a 2-arg invocation. Future PR that accidentally restores the 2-arg signature fails this test loudly.

`Order` is `SerializesModels`-compatible (the trait we already use), so the readonly Order property survives queue rehydrate cleanly. Worker pickup deserializes Order from `id` + class only ‚Äî no extra bytes on the queue payload.
