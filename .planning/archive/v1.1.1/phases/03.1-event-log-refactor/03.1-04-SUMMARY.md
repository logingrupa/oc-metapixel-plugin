## Threat Surface — No New Flags

All threats addressed by this plan were anticipated in the plan's `<threat_model>` block (T-3.1-18..24). No new trust boundaries, network endpoints, auth paths, file-access patterns, or schema changes introduced.

- **T-3.1-18 (Spoofing onMarkFired):** mitigated — server-side event_id match against the CAPI row; mismatch → reject + warn. UUIDv4 ~122 bits entropy makes guessing infeasible.
- **T-3.1-19 (Tampering payload event_time):** mitigated — EventLogWriter::record passes `(int) $obCapiRow->event_time` from the server's CAPI row, NOT a client-supplied value.
- **T-3.1-20 (Repudiation Pixel-fire claim):** mitigated — `fired_at` + `created_at` timestamps written on EventLog INSERT; UNIQUE prevents row mutation; rows append-only.
- **T-3.1-21 (Information Disclosure on mismatch log):** mitigated — Log::warning context carries `submitted_event_id_len` only; the value is never logged (locked by `test_onmarkfired_rejects_event_id_mismatch`'s dual-closure assertion).
- **T-3.1-22 (DoS onMarkFired flood):** accepted — Lovata.OrdersShopaholic rate-limits `/checkout/{slug}`; EventLogWriter is INSERT-only; UNIQUE constraint blocks duplicate work.
- **T-3.1-23 (Elevation without CAPI row):** mitigated — onMarkFired returns `['ok' => false]` when CAPI row absent; no EventLog row inserted.
- **T-3.1-24 (Tampering Twig sCustomDataJson|raw):** accepted — sCustomDataJson is pre-encoded server-side with the Phase-3 CR-01 JSON flags; `|raw` is safe because input is server-controlled and JSON-escaped at encode time.
