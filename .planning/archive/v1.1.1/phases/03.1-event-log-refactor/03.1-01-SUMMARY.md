## MIG-02 Lock Confirmation

`updates/drop_meta_purchase_columns_from_orders_table.php` lines 65 & 67 — verified by inspection:

```
65:                $obTable->dropIndex(self::INDEX_NAME);
67:            $obTable->dropColumn([self::COLUMN_ID, self::COLUMN_TIME]);
```

`dropIndex` (line 65) precedes `dropColumn` (line 67) inside the same `Schema::table` closure. SQLite cannot drop a column with a live index; Laravel 12's schema builder validates the ordering. The hermetic SQLite test harness in `MetapixelTestCase` runs migrations in this order without error.

## Decisions Made

- **version.yaml Option A** — preferred clean ledger preserving 1.0.1 entry rather than the roll-forward alternative (Option B in PATTERNS.md lines 1093-1096). Operators already on 1.0.1+ have the migration applied; the 1.0.1 → 1.1.0 trajectory needs the drop migration to fire.
- **Test-harness fixture minimalism** — `bootEventLogTable()` provisions UNIQUE + event_id index only, NOT the two secondary composite indices (secret_key, subject). Rationale: PATTERNS.md line 1056 — UNIQUE + event_id index are sufficient for every Phase 3.1 test. Production migration carries all four.
- **Down migration symmetry** — `DropMetaPurchaseColumnsFromOrdersTable::down()` mirrors the deleted source migration's `up()` body byte-for-byte (same `after('secret_key')` + `nullable()` + `index()` shape) so the v1.1.0 → v1.0.3 rollback restores the schema operators expected at v1.0.3 (T-3.1-01 mitigation honoured).
- **Class header docblock template** — each migration includes purpose, MIG-02 lock note (where applicable), reversibility statement, idempotency note, and `@see` cross-references to `BRIEF.md REFAC-NN` and `PATTERNS.md` analogs. Establishes the Phase 3.1 migration-comment convention for downstream waves.
