<objective>
Close the cross-context `site_id` divergence bug surfaced 2026-05-14 on
`new.nailscosmetics.lv` (orders 29802 + 29803): writer (admin /back
queue) persists `site_id=NULL` while reader (frontend /lv/checkout)
queries `where site_id=1` → `PurchasePixel::findEventLogRow(CAPI)`
returns null → gate fails → no Pixel block renders → browser never
tracks Purchase even though CAPI fires HTTP 2xx to Meta.

Root cause (BRIEF locked): `SiteResolver::getActiveSiteId()` reads
`SiteManager::instance()->getActiveSiteId()` which is request-context-
dependent. The Order itself already carries the authoritative answer
in `lovata_orders_shopaholic_orders.site_id` (Lovata v1.33 column,
populated by `OrderProcessor::arOrderData['site_id'] = Site::
getSiteIdFromContext()` at order-create time, immutable thereafter).

Fix shape (BRIEF locked — no replanning during execute):
  - REFAC-12: add `SiteResolver::forOrder(Order $obOrder): ?int` as
    the canonical Order-scoped resolver (reads `$obOrder->
    getAttribute('site_id')`). `getActiveSiteId()` stays for future
    non-Order subjects.
  - REFAC-13: rewire every Order-scoped call site (Watcher,
    SendCapiEvent, PurchasePixel) onto `forOrder`. Thread `?int
    $iSiteId` through `EventLogWriter::record` as an OPTIONAL 7th
    parameter (DRY — caller decides); writer becomes pure I/O.
  - REFAC-14: BACKFILL.sql repairs the 2 stranded production rows
    pre-deploy; STAGING-RUNBOOK Scenario 5 documents the admin-flip
    cross-context dance for operator verification; version.yaml bumps
    1.1.0 → 1.1.1 (patch — no schema, no public-API break).

MVP-TDD discipline (BRIEF lines 156-160): each behavior-adding task
lands its failing test as `test(03.1-07): RED — {method name}` BEFORE
the production-code commit that turns it green. Plan-checker trips if
RED commits are missing.

extensions/adjustments) + 1 test-base schema bump + BACKFILL.sql +
STAGING-RUNBOOK Scenario 5 + version.yaml + STATE + ROADMAP closure.
