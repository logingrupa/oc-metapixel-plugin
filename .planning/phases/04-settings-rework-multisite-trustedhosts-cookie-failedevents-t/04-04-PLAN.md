## Scope Rationale

This plan touches 15 files at the dimension-5 boundary because the FailedEvents UI bundle (ListController + 4 views + columns.yaml + Plugin::registerSettings hook + MetaClient extension + migration + model) is tightly coupled. Splitting into 04-04a (migration + model + MetaClient::fetchTestEventsStatus + Wave 0 tests) and 04-04b (controller + views + Plugin::registerSettings) was considered during revision iteration 1 and rejected: the controller and its views cannot ship independently — `config_list.yaml` references `_list_toolbar.htm` + `_graph_error.htm` + `columns.yaml` as a single rendering unit, the `onCheckDedup` handler depends on `MetaClient::fetchTestEventsStatus` AND the three new columns, and `Plugin::registerSettings` references the controller URL that does not exist until the controller file lands. Splitting would introduce churn (test scaffolds rerun twice, intermediate broken-state commits, duplicate `composer qa` runs) without reducing rollback footprint — if Task 2 acceptance fails, the entire bundle reverts together either way.

Risk accepted: Task 2 modifies 8 production files, so rollback footprint is large if acceptance fails. Mitigation: Task 0 ships 22+ Wave 0 RED test cases that exercise every code path BEFORE production code ships, so failures surface at unit-test boundary, not at integration. The plugin CLAUDE.md "Simple > clever" rule explicitly governs this decision — splitting a tightly-coupled MVC bundle into two sequential sub-plans is cleverness without payoff.
