## Areas Checked + Verified Clean

The following focus areas were audited and found correct (no findings):

- **Hungarian notation:** every local variable in the 5 modified production files uses `$ob`/`$ar`/`$i`/`$s`/`$b`/`$f`/`$m` prefix. `$obQuery`, `$iSubjectId`, `$iSiteId`, `$mId`, `$obException` — all consistent.
- **Explicit return types:** `SiteResolver::forOrder(): ?int`, `EventLogWriter::record(): bool`, `OrderStatusWatcher::alreadyDispatched(): bool`, `PurchasePixel::findEventLogRow(): ?EventLog`, `SendCapiEvent::raceFenceWon(): bool`. All declared.
- **Zero `assert(...)` usage:** `grep -rn 'assert(' classes/ components/` returns no production hits. PHPStan-level narrowing via `is_int` / `is_string && is_numeric` everywhere — Tiger-Style "no `assert`" honored.
- **Catch hygiene:** every `catch` in the touched files either rethrows (SendCapiEvent::handle MetaApiTransientException line 151) or carries an explicit `// silent: <reason>` comment (EventLogWriter::record line 115, SendCapiEvent::writeFailedEvent line 270, SiteResolver::getActiveSiteId line 87, OrderStatusWatcher::isPluginDisabled line 297, PurchasePixel::isDisabled line 366, PurchasePixel::onRun PayloadBuilder catch line 182).
- **SRP / DRY writer refactor:** EventLogWriter has ZERO `SiteResolver::` references. SendCapiEvent + PurchasePixel both pass `SiteResolver::forOrder($obSubject)` as the 7th arg — consistent.
- **is_numeric narrow correctness:** `SiteResolver::forOrder` narrow correctly rejects bool/array/object/null (confirmed via PHP probe). True/false → null. Empty string → null. Numeric strings → int. Float-shaped strings ("1.99") would coerce to int 1, but Order.site_id is an MySQL int column so the input vector is closed in practice.
- **NULL handling in Order.site_id `whereNull` vs `where(=, $iSiteId)`:** OrderStatusWatcher:279-283 and PurchasePixel:322-326 both correctly branch on `$iSiteId === null`. MySQL UNIQUE NULL-distinct semantic preserved.
- **BACKFILL.sql injection surface:** static SQL, no parameters bound, no operator-supplied substitution. MySQL backslash-escaping on `subject_type` literal (`'Lovata\\OrdersShopaholic\\Models\\Order'`) is correct under default mode (NO_BACKSLASH_ESCAPES mode would mismatch — operator should verify `sql_mode` does not include this flag, but that's a per-deploy concern not a code defect).
- **WHERE clause restrictiveness:** `WHERE el.site_id IS NULL AND o.site_id IS NOT NULL` — correctly NULL-only repair; never overwrites deliberately-NULL single-site rows. Idempotent (re-running post-deploy = 0 affected rows).
- **No new PII surfacing:** EventLog row columns unchanged. BACKFILL.sql writes only `site_id` int. STAGING Scenario 5 SQL probes show only operator-visible fields (id, channel, site_id, event_id, fired_at). No email / phone / IP escape via this phase.
- **Threat model T-3.1-30..34 mitigations:** Cross-context divergence (T-3.1-16 extension) closed at the contract level by `forOrder`. Audit trail preserved (no UPDATE/DELETE in writer). Append-only EventLog row + UNIQUE NULL-distinct + UUIDv4 paired event_id all intact.
- **RED → GREEN sequencing:** 5 RED commits land before 2 GREEN refactor commits per commit log. Plan-checker grep gate confirmed (`git log --oneline | grep '^[a-f0-9]* test(03.1-07): RED'` returns 5; `(fix|refactor)\(03.1-07\):` returns 4). MVP-TDD discipline honored.
- **Race conditions in cross-context tests:** `MultiSiteCrossContextTest.php` is deterministic — FIXED_EVENT_TIME constant + Carbon::setTestNow + explicit Config::set/null reset per test. No timing-dependent assertion.
- **Public-API backwards compat:** `EventLogWriter::record` 7th param `?int $iSiteId = null` is OPTIONAL — old 6-arg callers compile (silent-NULL hazard flagged as MED-02 but not a compile break). SendCapiEvent constructor 3rd arg `Order $obSubject` was a v1.1.0 BREAKING CHANGE (Phase 3.1-04), unchanged in 03.1-07.
